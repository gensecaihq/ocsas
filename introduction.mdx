## Abstract

AI agent systems represent a paradigm shift in human-computer interaction, converting model outputs into real-world actions through tools, persistent state, and control planes. This architectural evolution introduces a dominant risk class: **coercion combined with misconfiguration**. Empirical analysis demonstrates that severe security incidents in agentic systems frequently occur without exploiting traditional software vulnerabilities—they arise when untrusted inputs manipulate agents into unsafe tool invocations, or when operators inadvertently expose administrative surfaces, credentials, or browsing endpoints.

**OSSASAI** (Open Security Standard for Agentic Systems) establishes a vendor-neutral, globally-applicable security framework for AI agent runtimes. Drawing from established security research methodologies and international standards, OSSASAI provides a rigorous foundation for securing the emerging class of autonomous and semi-autonomous AI systems.

<Note>
**Normative Definition:** OSSASAI specifies baseline security controls, evidence requirements, and conformance verification procedures for AI agent systems that accept untrusted inputs and execute actions with real-world consequences via tool interfaces.
</Note>

## Motivation and Research Context

### The Agentic Security Gap

Contemporary AI security research has predominantly focused on model-level concerns: alignment, adversarial robustness, and output safety. However, the deployment of AI agents in production environments surfaces a distinct threat surface at the **runtime layer**—where models interface with tools, state, and external systems.

Prior work in this domain includes:
- **Prompt Injection Research** (Perez & Ribeiro, 2022; Greshake et al., 2023)
- **LLM Security Taxonomies** (OWASP Top 10 for LLMs, 2023)
- **Agent Safety Frameworks** (Kinniment et al., 2024)

OSSASAI synthesizes these research threads into an actionable control framework, addressing the gap between theoretical security analysis and operational deployment requirements.

### Design Principles

OSSASAI is constructed on five foundational principles derived from security engineering best practices:

<AccordionGroup>
  <Accordion title="Principle 1: Vendor Neutrality" icon="globe">
    OSSASAI maintains strict independence from any specific AI agent platform, model provider, or deployment architecture. Ecosystem-specific implementations are delegated to **profiles**—formal mappings that translate generic controls to concrete configurations. This separation ensures broad applicability while enabling platform-specific optimization.

    *Reference: ISO/IEC 27001 Annex A control structure methodology*
  </Accordion>
  <Accordion title="Principle 2: Threat-Centric Design" icon="crosshairs">
    Controls are derived from empirical threat analysis rather than prescriptive technology requirements. OSSASAI addresses documented failure modes including:
    - Prompt injection leading to tool misuse (Greshake et al., 2023)
    - Privilege escalation via over-permissioned tool interfaces
    - Control plane exposure and authentication bypass
    - Session boundary violations enabling cross-tenant data leakage
    - Secrets exfiltration through logs, memory, and telemetry channels

    *Reference: STRIDE threat modeling (Microsoft SDL)*
  </Accordion>
  <Accordion title="Principle 3: Verifiability" icon="check-double">
    Every control includes formally specified verification procedures. Controls without testable acceptance criteria are excluded from the normative specification. This enables:
    - Automated compliance verification
    - Reproducible security assessments
    - Continuous conformance monitoring

    *Reference: NIST SP 800-53A Assessment Procedures*
  </Accordion>
  <Accordion title="Principle 4: Tiered Maturity Model" icon="stairs">
    Three assurance levels (L1–L3) provide graduated security requirements aligned with deployment risk profiles. This tiered approach:
    - Reduces barrier to entry for lower-risk deployments
    - Establishes clear progression paths for security maturation
    - Enables risk-proportionate control implementation

    *Reference: CMMI maturity level framework; OWASP ASVS level structure*
  </Accordion>
  <Accordion title="Principle 5: Actionable Guidance" icon="hammer">
    OSSASAI provides implementation guidance, not merely requirements. Each control includes:
    - Remediation procedures
    - Platform-agnostic configuration examples
    - Evidence collection specifications

    *Reference: CIS Benchmark implementation methodology*
  </Accordion>
</AccordionGroup>

## OSSASAI Top 10: Primary Failure Modes

Based on empirical analysis of agent system incidents and security research literature, OSSASAI prioritizes the following failure modes:

| Rank | Failure Mode | Description | Primary Controls |
|:----:|--------------|-------------|------------------|
| 1 | **Prompt Injection → Tool Misuse** | Adversarial inputs coerce agents into unintended tool invocations | TB-01, TB-02, ID-02 |
| 2 | **Over-Privileged Tools** | Tool interfaces granted excessive capabilities violating least privilege | TB-01, TB-02, TB-04 |
| 3 | **Control Plane Exposure** | Administrative interfaces accessible to unauthorized parties | CP-01, CP-02, CP-03 |
| 4 | **Session Boundary Collapse** | Cross-user context leakage and instruction carryover | ID-02, ID-03 |
| 5 | **Secrets Exfiltration** | Credential leakage via logs, memory, telemetry, or backups | LS-01, LS-02 |
| 6 | **Unsafe External Connectivity** | SSRF-like patterns and data exfiltration through tool interfaces | TB-03, TB-04 |
| 7 | **Extension Supply Chain** | Unreviewed plugins and skills introducing malicious functionality | SC-01, SC-02 |
| 8 | **Memory Poisoning** | Retrieval-augmented attacks and instruction smuggling | LS-03 |
| 9 | **Insufficient Auditability** | Missing action ledgers and approval traceability | TB-02, LS-04 |
| 10 | **Configuration Drift** | Absence of posture verification enabling security regression | CP-01, CP-02 |

## Assurance Level Summary

| Level | Designation | Target Deployment | Risk Profile |
|:-----:|-------------|-------------------|--------------|
| **L1** | Local-First Baseline | Single operator, local administration, minimal network exposure | Hobby/personal use, developer workstations |
| **L2** | Network-Aware | Remote access via LAN/VPN/tailnet, team usage, remote control plane | Small teams, internal tools, controlled environments |
| **L3** | High-Risk Runtime | Tool-rich, plugin-heavy, multi-tenant, broad external connectivity | Enterprise deployment, regulated industries, high-value targets |

## Profile System: OCSAS for OpenClaw

**OCSAS** (OpenClaw Security Assurance Standard) serves as the reference implementation profile, demonstrating the OSSASAI profile mechanism:

```yaml
# Profile metadata
profile_id: OSSASAI-PROFILE-OPENCLAW-OCSAS-0.1
platform: OpenClaw
version: 0.1.0

# Example control mapping
mappings:
  OSSASAI-CP-01:
    config_key: gateway.bind
    default: "127.0.0.1:18789"
    verification: "openclaw security audit --deep"

  OSSASAI-ID-02:
    config_key: session.dmScope
    options: ["per-channel-peer", "per-account-channel-peer"]
    verification: "openclaw test session-isolation"
```

See the [OpenClaw Profile (OCSAS)](/profiles/openclaw) for complete implementation guidance.

## Quick Start

<Steps>
  <Step title="Determine Assurance Level">
    Review [assurance level criteria](/spec/assurance-levels) to identify appropriate target level based on deployment risk profile.
  </Step>
  <Step title="Select Implementation Profile">
    For OpenClaw deployments, use [OCSAS](/profiles/openclaw). For other platforms, implement controls per the generic [specification](/spec/overview).

    ./ossasai-audit.sh --level L2

  <Step title="Remediate Findings">
    Address identified gaps using [implementation guides](/implementation/quickstart).
  </Step>
  <Step title="Generate Evidence">
    Collect [required artifacts](/compliance/evidence-collection) for compliance verification.
  </Step>
</Steps>

## Conformance and Certification

### Self-Assessment

Organizations may conduct self-assessment using:
- OSSASAI audit tooling (`ossasai-audit.sh`)
- Evidence collection procedures
- Conformance report templates

### Third-Party Verification

For L3 deployments and formal certification, OSSASAI supports:
- Independent security assessments
- Formal verification evidence review
- Continuous compliance monitoring

## Governance and Development

### Working Group

OSSASAI is maintained by an open community working group. Participation is open to:
- Security researchers
- Platform developers
- Enterprise security practitioners
- Academic institutions

### Contribution Process

1. **Issues**: Report gaps, propose new controls, identify errors
2. **Proposals**: Submit formal control proposals with threat justification
3. **Reviews**: Participate in community review of proposed changes
4. **Profiles**: Develop and submit ecosystem-specific profiles
